VMware: STEPS TO CREATE THE VIRTUAL MACHINES

A clear reference for setting up virtual machines (Windows Server 2016 labs)

1. Typical (Recommended) Setup

Guided wizard using VMware defaults — fast and easy for most users.

Quick
  1. Open VMware Workstation → File > New Virtual Machine.
  2. Choose Typical (recommended).
  3. Select installation media:
    • Install from installer disc (physical DVD)
    • Install from installer disc image (ISO)
    • Install OS later
  4. Choose guest OS type (Windows, Linux, etc.).
  5. Name the VM and choose its location.
  6. Specify disk size and storage method (single file or split).
  7. Review the summary → Finish.
👉 Note: Typical is best when you want a quick VM and are learning server basics — installs fast with sensible defaults.

2. Custom (Advanced) Setup

Full control over virtual hardware and settings — for experienced users or special needs.

Advanced
  1. Open VMware Workstation → File > New Virtual Machine.
  2. Select Custom (advanced).
  3. Configure VM hardware compatibility (choose hardware version).
  4. Select installation media (disc, ISO, or install OS later).
  5. Define OS type and version.
  6. Adjust VM name, storage location, and virtual disk details.
  7. Customize hardware: CPU cores, memory, network type (NAT, bridged, host-only), SCSI controller, USB compatibility, etc.
  8. Review settings → Finish.
👉 Note: Use Custom when you need to fine-tune VM behavior (e.g., specific CPU topology, pass-through devices, or special network setups).

2. Custom (Advanced) Setup — Detailed Explanation

Step 3: Configure VM Hardware Compatibility

What it means:

Compatibility is the virtual hardware version VMware emulates. Newer versions provide support for newer features (TPM 2.0, newer DirectX, etc.).

Example:
  • Workstation 17.x — supports latest OS (Windows 11, modern Linux), TPM 2.0, newer DirectX.
  • Workstation 12.x — older features, suitable for legacy OS like Windows XP.

Real-world analogy: choosing compatibility is like choosing whether your physical PC is "modern" (USB-C, PCIe 5.0) or "older" (USB 2.0).

Step 4: Select Installation Media

Options include:

  • Installer disc (physical)
  • Installer disc image (.iso) — most common
  • I will install the OS later — creates blank VM

Step 5: Define OS Type and Version

VMware uses this to set recommended defaults (EFI vs BIOS, disk defaults, integration tools).

Step 6: VM Name, Location & Disk

Choose a clear name (e.g., Win2016-DC) and a storage location for the VM files. Disk options: set max size and choose single or split files.

Step 7: Customize Hardware

Common hardware items to tune:

  • Processors: sockets and cores. Tip: leave some host cores free.
  • Memory (RAM): allocate depending on host capacity (e.g., host 16 GB → give Windows VM 4–8 GB).
  • Network: NAT, Host-only, or Bridged — details below.
Quick CPU example

Host: Intel i7 (4 cores / 8 threads) — good VM assignment: 2 cores (leave host resources).

Network Types — Student Recommendations

Choose based on safety, internet need, and realism.

1. NAT — Best for Beginners

How it works: VM shares host IP for internet access.

Use case: Download updates, learn AD without exposing VM to LAN.

Pros: Safe and simple.

Cons: Other devices cannot reach VM directly.

2. Host-Only — For Isolated Labs

How it works: VM communicates only with the host.

Use case: Active Directory domain labs.

Pros: Safe, prevents accidental changes to real network.

Cons: No internet unless combined with NAT.

3. Bridged — For Realistic Practice

How it works: VM gets an IP from the same router as the host.

Use case: Let others on LAN access your VM.

Pros: Real-world simulation.

Cons: Can disrupt real network services — ask instructor first.

Suggested student flow
  1. Start with NAT for general learning.
  2. Use Host-only to build an isolated AD lab (DC + clients).
  3. Use Bridged only with instructor approval for LAN testing.
Printable Tips
  • Give descriptive VM names (Win2016-DC, Client01).
  • Snapshot before experimenting with domain or network changes.
  • Don't allocate all host RAM/CPU to VMs.
