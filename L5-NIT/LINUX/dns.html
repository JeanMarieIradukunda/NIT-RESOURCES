🌐 DNS Configuration – Padri Vjeko Centre TSS

1️⃣ What Is DNS — and Why Does It Matter in Your School?

DNS (Domain Name System) is like the phonebook of the internet. It translates names (like student.padri-vjeko.tss) into IP addresses (like 192.168.10.15) that computers understand.

In your school setting:

  • Instead of remembering IPs like 192.168.10.10, you can just type server.nit.local or portal.padri-vjeko.tss.
  • It helps organize internal networks for different departments (NIT, FAD, WOT, BDC).

(d) DNS Records — Complete & Friendly Guide

Below are the common (and some advanced) DNS record types you’ll meet while administering DNS at Padri Vjeko Centre TSS. Each entry includes a plain-language description, an example, and a short note on why it matters.

🛠️ SOA (Start of Authority)

The SOA record appears once per zone and defines the zone’s authoritative information: the primary name server for the zone, the admin email, and important timing values (serial, refresh, retry, expire, negative TTL).

Example (zone header):

$TTL 604800
@   IN  SOA  server.padri-vjeko.tss. admin.padri-vjeko.tss. (
        2025110301 ; Serial (YYYYMMDDnn)
        604800     ; Refresh
        86400      ; Retry
        2419200    ; Expire
        604800 )   ; Negative TTL

🔎 Why it matters: The serial informs secondary (slave) servers when the zone has changed; increment it when you update records.

📡 NS (Name Server)

Defines which DNS servers are authoritative for the zone. You typically have at least one NS record (often more for redundancy).

Example: padri-vjeko.tss. IN NS server.padri-vjeko.tss.

Note: For internet-visible domains you add multiple NS records (distributed across networks); for internal setups a single reliable NS may suffice.

🌐 A (Address)

Maps a hostname to an IPv4 address — the bread-and-butter record for hosting services on IPv4.

Example: server.padri-vjeko.tss → 192.168.10.1

Use: Point the school’s web or file server hostname to its IPv4 address.

🔷 AAAA (IPv6 Address)

Same as A but for IPv6 addresses. Use this if parts of your network or services use IPv6.

Example: server.padri-vjeko.tss → 2001:db8:abcd:001::1

Note: Not required for purely IPv4 networks, but useful if you plan IPv6 readiness.

🔗 CNAME (Canonical Name / Alias)

Creates an alias name that points to a canonical name. Don’t use CNAME at the zone apex (root) — use A/AAAA there.

Example: www.padri-vjeko.tss → server.padri-vjeko.tss

Use: Friendly aliases like www or portal that reuse the same server records.

✉️ MX (Mail Exchange)

Directs email for a domain to a mail server. MX records include a priority; lower numbers = higher priority.

Example: padri-vjeko.tss. IN MX 10 mail.padri-vjeko.tss.

Tip: Use a PTR (reverse) for the mail server IP and proper SPF/TXT to reduce email delivery problems.

🔁 PTR (Pointer / Reverse Lookup)

Maps an IP address back to a hostname. Typically stored in a reverse zone (in-addr.arpa for IPv4).

Example: 192.168.10.1 → server.padri-vjeko.tss

Why: Useful for logging, diagnostics, and mail server reputation checks.

📝 TXT (Text Record)

Free-form text attached to a name. Widely used for domain verification, notes, and security policy records like SPF and DKIM.

Examples:

padri-vjeko.tss. IN TXT "v=spf1 ip4:192.168.10.0/24 -all"
padri-vjeko.tss. IN TXT "google-site-verification=XYZ123"

Note: SPF is usually stored as a TXT record (legacy practice). For stricter DMARC/DKIM, add those TXT records too.

📛 SPF (Sender Policy Framework) — stored in TXT

Declares which hosts/IPs are allowed to send email for your domain. Usually a TXT record that begins with v=spf1.

Example: padri-vjeko.tss. IN TXT "v=spf1 ip4:192.168.10.0/24 -all"

Tip: Keep SPF rules simple and test after publishing (too-strict SPF can block legitimate mail).

⚙️ SRV (Service Record)

Used to locate services (like SIP, XMPP, Active Directory) by specifying target host and port. SRV records are common in service discovery.

Example: _sip._tcp.padri-vjeko.tss. IN SRV 10 5 5060 sipserver.padri-vjeko.tss.

Use: If you run VoIP, chat servers, or AD services that need service discovery, configure SRV entries.

🔒 CAA (Certificate Authority Authorization)

Specifies which Certificate Authorities (CAs) are allowed to issue TLS/SSL certificates for your domain — a security control.

Example: padri-vjeko.tss. IN CAA 0 issue "letsencrypt.org"

Why: Helps prevent unauthorized certificate issuance for your internal or external services.

🧾 Quick Notes & Best Practices
  • TTL (Time To Live): Controls how long records are cached by clients and resolvers. Lower during changes (e.g., 300) and higher for stable records.
  • Serial numbers: Increment the SOA serial whenever you change zone files so secondaries pick up updates.
  • Zone apex caution: Don’t put a CNAME record at the zone apex (e.g., padri-vjeko.tss). Use A/AAAA instead.
  • Security: Use PTR for mail servers, SPF/DKIM/DMARC to protect email, and CAA for certificate control.
  • Documentation: Keep a small change-log in your /etc/bind/ directory or a separate admin doc for each update.

(d) DNS Records Explained

In DNS (Domain Name System), different types of records define how names are translated into IP addresses, how mail is delivered, and how reverse lookups are performed. Below are the main record types you’ll configure at Padri Vjeko Centre TSS.

🔹 A Record (Address Record)

This record maps a domain name to an IPv4 address. It’s like a phonebook entry linking a person’s name to their phone number.

Example: server.padri-vjeko.tss → 192.168.10.1

🏫 At Padri Vjeko Centre TSS: The school’s main file server named server.padri-vjeko.tss can be accessed internally using the IP 192.168.10.1.

🔹 CNAME Record (Canonical Name)

This record creates an alias for another domain name. It helps when you want multiple names to point to the same server.

Example: www.padri-vjeko.tss → server.padri-vjeko.tss

🖥️ In the school’s case: When students type www.padri-vjeko.tss in a browser, it automatically connects them to the same server as server.padri-vjeko.tss.

🔹 MX Record (Mail Exchange)

Defines which server handles email delivery for your domain. It tells other mail systems where to send messages.

Example: mail.padri-vjeko.tss → 192.168.10.2

📧 In our school setup: The mail server mail.padri-vjeko.tss receives emails for teachers and students using addresses like teacher@padri-vjeko.tss.

🔹 PTR Record (Pointer Record)

Used for reverse DNS lookup — it maps an IP address back to a domain name. This is often used for verifying mail servers and improving network trust.

Example: 192.168.10.1 → server.padri-vjeko.tss

🔁 At Padri Vjeko Centre TSS: When checking logs or receiving emails, the server at 192.168.10.1 can be identified as server.padri-vjeko.tss.

📁 Main BIND9 Configuration Files

Located in /etc/bind/ after installation.

1. named.conf – Main Configuration

Acts as the “control center” and includes other files.

2. named.conf.options – Global Options

Sets forwarders, listen-on IPs, recursion rules. Example: 8.8.8.8

3. named.conf.local – Local Zones

zone "padri-vjeko.tss" {
    type master;
    file "/etc/bind/db.padri-vjeko.tss";
};
zone "10.168.192.in-addr.arpa" {
    type master;
    file "/etc/bind/db.192.168.10";
};

4. Zone Files (db.*)

a) Forward Lookup Zone – Maps names to IPs

nit IN A 192.168.10.20
fad IN A 192.168.10.40
wot IN A 192.168.10.60
bdc IN A 192.168.10.80

b) Reverse Lookup Zone – Maps IPs to names

20 IN PTR nit.padri-vjeko.tss.
40 IN PTR fad.padri-vjeko.tss.
60 IN PTR wot.padri-vjeko.tss.
80 IN PTR bdc.padri-vjeko.tss.

5. rndc.key

Used for secure remote DNS control.

6. /var/cache/bind/

Stores cached lookups for faster responses.

7. named.conf.default-zones

Contains default localhost zones.

🌐 DNS Server Zones — Complete Explanation

🧱 What Is a DNS Zone?

A zone is a portion of the DNS database that stores records for a domain or subdomain.

🧭 Main Types of DNS Zones

🟩 A. Forward Lookup Zone

Resolves names → IP addresses. File: /etc/bind/db.padri-vjeko.tss

🟨 B. Reverse Lookup Zone

Resolves IP → name. File: /etc/bind/db.192.168.10

🟦 C. Primary (Master) Zone

Main copy of zone data maintained on the Ubuntu DNS server.

🟪 D. Secondary (Slave) Zone

Read-only backup copy of primary zone.

🟧 E. Stub Zone

Holds only basic records (NS, SOA) of another zone for linking networks.

🟫 F. Forward Zone (Forwarder)
forwarders {
    8.8.8.8;
    1.1.1.1;
};

Used to forward queries to external DNS servers.
