Linux and DNS Notes

1. Linux File System Hierarchy

The Linux file system hierarchy is like the “map” of all files and directories in Linux. Everything starts from the root directory, represented by /. All files and directories branch out from this root.

Here’s a simple description of the most important directories:

DirectoryDescription (Human-friendly)
/Root directory. Everything starts here.
/binEssential binaries (programs) for all users, like ls, cp.
/sbinSystem binaries used by administrators (root), like ifconfig.
/etcConfiguration files for system and services, like network configs.
/homeUsers’ personal directories. Each user gets /home/username.
/rootHome directory for the root (administrator) user.
/varVariable data, like logs (/var/log) or databases.
/tmpTemporary files that get deleted on reboot.
/usrApplications and utilities for users. /usr/bin for programs, /usr/lib for libraries.
/libShared libraries required for programs to run.
/optOptional software packages (like third-party apps).
/mediaMount points for removable media like USB drives.
/mntTemporary mount points for disks or other file systems.
💡 Tip: Think of / as the “root of a tree,” and all directories are branches.

2. Linux File System Types / Categories

Linux supports multiple file system types. A file system is the method Linux uses to store and organize files on a disk.

Common Linux File System Types:

  1. Ext4 (Fourth Extended Filesystem)
    • Definition: Ext4 is the default Linux filesystem used in Ubuntu. It is reliable, fast, and supports modern features like journaling.
    • Key Features:
      • Supports large files (up to 16 TB) and large volumes (up to 1 EB).
      • Journaling keeps track of changes to prevent data loss during crashes.
      • Backward compatible with Ext3.
      • Delayed allocation for better performance.
    • Pros:
      • Fast and stable for most workloads.
      • Supports very large files and partitions.
      • Widely supported on Linux.
    • Cons:
      • Lacks some advanced features of newer filesystems like snapshots.
      • Not ideal for extremely high-performance enterprise storage.
    • Use Cases:
      • Default filesystem for Ubuntu installations.
      • General-purpose servers and desktops.
  2. Ext3 (Third Extended Filesystem)
    • Definition: Ext3 is an older version of Ext4 with journaling. It was the default filesystem for Linux before Ext4.
    • Key Features:
      • Supports journaling to protect against data corruption.
      • Compatible with Ext2 and Ext4.
      • Supports large volumes up to 16 TB.
    • Pros:
      • Simple and reliable.
      • Easy to upgrade to Ext4.
      • Well-tested and stable.
    • Cons:
      • Slower than Ext4 for large files.
      • Limited performance optimizations.
    • Use Cases:
      • Legacy Linux systems.
      • Systems that require high stability over performance.
  3. XFS
    • Definition: XFS is a high-performance, journaling filesystem optimized for large files and scalable systems.
    • Key Features:
      • Supports extremely large files and volumes.
      • Uses journaling to maintain data integrity.
      • Excellent performance for parallel I/O and multi-threaded workloads.
    • Pros:
      • Very fast for large files and high-performance servers.
      • Scales well for enterprise storage solutions.
    • Cons:
      • Slightly more complex to manage.
      • Not ideal for small, low-capacity drives.
    • Use Cases:
      • File servers and database servers.
      • Systems handling large multimedia files.
  4. Btrfs (B-tree Filesystem)
    • Definition: Btrfs is a modern Linux filesystem designed for advanced storage features like snapshots, compression, and error detection.
    • Key Features:
      • Supports snapshots and rollbacks.
      • Built-in data integrity checks and repair.
      • Supports compression and subvolumes.
      • Scales to large volumes.
    • Pros:
      • Advanced features for storage management.
      • Protects data against corruption.
      • Flexible and scalable.
    • Cons:
      • Slightly less mature than Ext4 in some distributions.
      • Performance can vary with heavy workloads.
    • Use Cases:
      • Advanced storage systems.
      • Backup and recovery setups.
      • Systems needing snapshots and versioning.
  5. FAT32 / NTFS
    • FAT32:File Allocation Table 32
    • NTFS:New Technology File System
    • Definition: FAT32 is an older, simple filesystem for smaller drives and cross-platform use. NTFS is a modern Windows filesystem with support for large files and advanced features.
    • Key Features:
      • FAT32: Maximum file size 4 GB, compatible across almost all systems.
      • NTFS: Supports very large files, permissions, encryption, and journaling.
    • Pros:
      • FAT32: High compatibility, simple to use.
      • NTFS: Secure, supports large files, journaling protects against crashes.
    • Cons:
      • FAT32: Cannot store files larger than 4 GB, no permissions.
      • NTFS: Less compatible with Linux/macOS without extra drivers.
    • Use Cases:
      • FAT32: USB drives, SD cards, external drives shared across OS.
      • NTFS: Windows system drives, external drives needing security and large file support.

Categories of Linux file systems:

  • Local file systems: Stored on your own disks (e.g., Ext4, XFS).
  • Network file systems: Stored on remote servers (e.g., NFS, SMB/CIFS).
  • Special file systems: For devices or kernel info (e.g., /proc, /sys).
💡 Tip: If you are just using Ubuntu for a server, most internal drives use Ext4.

3. Creating and Managing Directories and Files

Creating Directories

# Create a single directory
mkdir my_directory

# Create nested directories (parent/child)
mkdir -p projects/python

Listing Directories and Files

# Show files and directories in current folder
ls

# Show detailed info with permissions, size, date
ls -l

# Show hidden files (starting with .)
ls -a

Creating Files

# Create an empty file
touch myfile.txt

# Create a file and add content
echo "Hello Ubuntu Server" > hello.txt

# Create a file using a text editor
nano myfile.txt   # Opens nano editor

Managing Files and Directories

# Rename or move a file/directory
mv oldname.txt newname.txt
mv file.txt /home/user/Documents/

# Copy files/directories
cp file.txt backup_file.txt
cp -r my_directory /home/user/backup/   # -r for recursive (directories)

# Delete files/directories
rm file.txt
rm -r my_directory   # -r for directories

Checking File Information

# Check file type
file myfile.txt

# Show file size
du -h myfile.txt

# Show disk usage of a directory
du -sh my_directory
💡 Tip: Linux is case-sensitive. File.txt and file.txt are different.

4. Users in Linux

A user is an account that can log into the system. Each user has certain attributes:

  • Login Name: The username used to log in, e.g., iradukunda.
  • Encrypted Password: Linux stores passwords in an encrypted form in /etc/shadow.
  • UID (User ID): Unique number assigned to each user.
    • System users usually have UIDs < 1000.
    • Normal users have UIDs ≥ 1000.
  • GID (Primary Group ID): Each user belongs to a primary group, stored as a GID.
  • User’s Default Shell: The command-line shell assigned to the user, e.g., /bin/bash.

Creating a User

# Create a new user with home directory and default shell
sudo useradd -m -s /bin/bash username

# Set a password for the user
sudo passwd username

Options Explained:

  • -m → Create a home directory /home/username
  • -s /bin/bash → Set default shell as Bash

Viewing User Information

# Show user info from /etc/passwd
grep username /etc/passwd

# Show encrypted password from /etc/shadow (requires root)
sudo grep username /etc/shadow

5. Groups in Linux

A group is a collection of users, used to manage permissions. Each group has:

  • Group Name
  • GID (Group ID)
  • Group Members
  • Group Password (optional)
  • Group Ownership
  • Group Permissions
  • Group Expiration Date (optional)

Creating a Group

# Create a new group
sudo groupadd groupname

# Create a group with a specific GID
sudo groupadd -g 2000 mygroup

Managing Users in Groups

# Add a user to a group
sudo usermod -aG groupname username

# Check group membership
groups username

# Example:
sudo usermod -aG sudo iradukunda
groups iradukunda
# Output: iradukunda : iradukunda sudo

Removing a User from a Group

sudo gpasswd -d username groupname

Viewing Group Information

# List all groups
cat /etc/group

# Show specific group info
grep groupname /etc/group

Deleting Users and Groups

# Delete a user and their home directory
sudo userdel -r username

# Delete a group
sudo groupdel groupname

Advanced: User Expiration and Password Policies

# Set account expiration date
sudo chage -E 2025-12-31 username

# Lock a user account
sudo usermod -L username

# Unlock a user account
sudo usermod -U username
💡 Tip: Always use groups to manage permissions efficiently instead of assigning file access to individual users.

6. Linux File and Directory Permissions

Permission Types

PermissionSymbolMeaning
ReadrCan view the contents of the file or list the files in a directory.
WritewCan modify or delete the file; can create/delete files in a directory.
ExecutexCan run the file as a program; can enter (cd) into a directory.

User Categories

CategorySymbolWho it applies to
OwneruThe user who owns the file.
GroupgUsers belonging to the file’s group.
OthersoEveryone else on the system.

Viewing Permissions

ls -l filename

Example Output:

-rwxr--r-- 1 iradukunda staff 123 Nov 10 13:00 file.txt

Explanation:

  • - → File type (d for directory, - for regular file)
  • rwx → Owner permissions: read, write, execute
  • r-- → Group permissions: read only
  • r-- → Others permissions: read only
  • 1 → Number of links
  • iradukunda → Owner
  • staff → Group
  • 123 → File size in bytes
  • Nov 10 13:00 → Last modified date
  • file.txt → File name

Setting Permissions

A. Symbolic Notation

Format: chmod [who][operation][permissions] filename

OperationSymbolMeaning
Add+Add a permission
Remove-Remove a permission
Set=Set exact permission
# Add execute permission for owner
chmod u+x file.txt

# Remove write permission for group
chmod g-w file.txt

# Set read and write for owner, read for group and others
chmod u=rw,g=r,o=r file.txt

B. Octal Notation

Permissions can also be represented as a 3-digit number, each digit representing owner, group, others.

SymbolicOctal
---0
--x1
-w-2
-wx3
r--4
r-x5
rw-6
rwx7
# Owner rwx, group r-x, others r--
chmod 754 file.txt

Changing Ownership

# Change file owner
sudo chown newowner file.txt

# Change file group
sudo chgrp newgroup file.txt

# Change owner and group at once
sudo chown newowner:newgroup file.txt

Testing Permissions

# Switch to another user
su - testuser

# Try reading, writing, executing a file
cat file.txt        # Test read
echo "hello" >> file.txt   # Test write
./file.sh           # Test execute
💡 Tip: Directories need execute permission (x) to allow users to enter (cd) them. Files need execute permission (x) to run scripts or programs.

7. Installation of DNS Server Package

# Update package index
sudo apt update

# Install BIND9 and related utilities
sudo apt install bind9 bind9utils bind9-doc -y

# Enable and start the BIND9 service
sudo systemctl enable bind9
sudo systemctl start bind9

# Check status
sudo systemctl status bind9
💡 Tip: bind9utils includes helpful tools like dig and nslookup. bind9-doc contains documentation.

8. File Configuration

BIND9 uses several configuration files, mainly stored in /etc/bind/:

FilePurpose
/etc/bind/named.confMain configuration file, includes other files.
/etc/bind/named.conf.optionsGlobal server options, e.g., forwarders.
/etc/bind/named.conf.localLocal zone definitions.
/etc/bind/named.conf.default-zonesDefault system zones like localhost.
/var/cache/bind/DNS cache (temporary files).
/var/lib/bind/Optional storage for dynamic zones.

Example: Configuring Forwarders

sudo nano /etc/bind/named.conf.options
options {
    directory "/var/cache/bind";

    recursion yes;
    allow-query { any; };

    forwarders {
        8.8.8.8;    # Google DNS
        8.8.4.4;
    };

    dnssec-validation auto;
    auth-nxdomain no;
};
💡 Tip: Recursion allows the server to query other servers on behalf of clients. Forwarders are DNS servers to query if this server doesn’t know the answer.

9. DNS Zones Configuration

Step 1: Define Zone in named.conf.local

sudo nano /etc/bind/named.conf.local
zone "padri-vjeko.tss" {
    type master;
    file "/etc/bind/zones/db.padri-vjeko.tss";
};

zone "0.168.192.in-addr.arpa" {
    type master;
    file "/etc/bind/zones/db.192.168.0";
};

Forward zone maps domain names to IPs, reverse zone maps IPs to domain names.

Step 2: Create Zone Files

NS zone files are the files that a DNS server uses to store information about a domain.

Example of a Human-Friendly Zone File

$TTL 86400
@       IN  SOA   ns1.padri-vjeko.tss. admin.padri-vjeko.tss. (
                 2025111001 ; serial
                 604800     ; refresh
                 86400      ; retry
                 2419200    ; expire
                 604800 )   ; negative cache

; Name servers
@       IN  NS    ns1.padri-vjeko.tss.

; A records
ns1     IN  A     192.168.0.10
www     IN  A     192.168.0.20
mail    IN  A     192.168.0.30

; MX record
@       IN  MX    10 mail.padri-vjeko.tss
💡 Think of a zone file like a phonebook: Each domain/subdomain is like a person’s name. Each record is like their phone number, email, or address. The DNS server is the phone operator, using the zone file to connect people to the right addresses.

Forward Zone File Example: db.padri-vjeko.tss

$TTL    604800
@       IN      SOA     ns1.padri-vjeko.tss. admin.padri-vjeko.tss. (
                        2025111001 ; Serial
                        604800     ; Refresh
                        86400      ; Retry
                        2419200    ; Expire
                        604800 )   ; Negative Cache TTL
;
@       IN      NS      ns1.padri-vjeko.tss.
ns1     IN      A       192.168.0.10
www     IN      A       192.168.0.20
mail    IN      A       192.168.0.30

Reverse Zone File Example: db.192.168.0

$TTL    604800
@       IN      SOA     ns1.padri-vjeko.tss. admin.padri-vjeko.tss. (
                        2025111001 ; Serial
                        604800     ; Refresh
                        86400      ; Retry
                        2419200    ; Expire
                        604800 )   ; Negative Cache TTL
;
@       IN      NS      ns1.padri-vjeko.tss.
10      IN      PTR     ns1.padri-vjeko.tss.
20      IN      PTR     www.padri-vjeko.tss.
30      IN      PTR     mail.padri-vjeko.tss.

Set Permissions for Zone Files

sudo chown root:bind /etc/bind/zones/db.*
sudo chmod 640 /etc/bind/zones/db.*

Test DNS Configuration

# Check BIND Syntax
sudo named-checkconf
sudo named-checkzone padri-vjeko.tss /etc/bind/zones/db.padri-vjeko.tss
sudo named-checkzone 0.168.192.in-addr.arpa /etc/bind/zones/db.192.168.0

# Restart BIND
sudo systemctl restart bind9
sudo systemctl status bind9

# Test DNS Queries using dig
dig @localhost www.padri-vjeko.tss
dig @localhost mail.padri-vjeko.tss
dig -x 192.168.0.20

# Test DNS Queries using nslookup
nslookup www.padri-vjeko.tss 127.0.0.1
💡 Tip: Always increment the Serial number in zone files after changes. Forwarders help your DNS resolve external domains like google.com. Use dig +trace domain.com to debug DNS resolution step by step.
